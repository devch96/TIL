# 코딩에서의 협업

---------

## 코드를 작성하는 행위

### 프로그래밍 중 이루어지는 다양한 활동

- 프로그램을 작성하다 보면 여러 종류의 활동이 이루어진다.

#### 검색

- 코드베이스를 살펴보고 특정 정보를 검색하는 작업
  - 해결해야 하는 버그의 정확한 위치나 특정 메서드의 모든 호출 혹은 변수가 초기화되는 위치 등
- 중단점과 디버거를 사용하기도 하고 출력문으로 확인하기도 한다.
- 검색은 STM에 무리를 가함
- 코드 내에서 검색할 때 특정 코드를 살펴본 이유를 주석문으로 작성해놓으면 도움이 됨
  - 페이지 클래스 초기화에 관련될 수 있다고 생각해 이 메서드를 확인함

#### 이해

- 코드를 읽고 실행해봄으로써 그 기능을 이해한다.
  - 코드를 읽고 실행하는 것 외에도 코드가 작동하는 방식을 더 잘 이해하기 위한 테스트 코드의 실행도 포함
- 작업 기억 공간에 무리를 가하기에 코드를 이해하기 쉽게 리팩터링하는 것이 도움이 됨

#### 전사

- 단순히 코딩하는 활동
  - 코드베이스에 추가 또는 변경할 내용에 대한 구체적인 계획이 있을 때 단지 그 계획을 수행하는 것
- 문법 구조를 떠올릴 수 있어야 하기 때문에 LTM에 부하를 유발

#### 증가

- 검색, 이해, 전사가 합쳐진 활동
- 코드를 추가할 위치를 검색하는 것, 어디에 코드를 추가할지 그리고 어떻게 할지 이해하기 위해 기존 코드를 파악하는 것, 그리고 아이디어를
문법으로 실제로 전사하는 활동이 모두 포함됨
- 3가지 기억 체계에 모두 부하를 초래할 수 있음

#### 탐구

- 코드를 사용하여 스케치하는 것
- 무엇을 해야 할지 분명한 생각이 떠오르지 않을 때, 프로그래밍을 통해 문제의 도메인과 사용해야 하는 프로그래밍 구성 요소를 명확히 알 수 있음
- 다른 활동에 의존하기 때문에 3가지 기억 공간 모두 어려움을 겪음

### 프로그래머의 업무 중단

- 슬랙 등의 메시지 앱으로 인해 업무 중단이 더 일반적이 되었다.

#### 중단 후에는 어떻게 되는가?

- 원래 하던 코딩 작업으로 돌아가기 위해 작업 기억 공간이 일을 더 한다.
- 프로그래밍 작업을 재개하기 전 코드의 여러 위치를 찾아다님

#### 중단에 잘 대비하는 방법

- 정신 모델 저장
  - 모델의 일부가 코드와 별도로 저장되어 있으면 정신 모델을 빠르게 되찾을 수 있다.
    - 주석문 등
- 미래 기억 향상
  - 미래에 무언가를 할 것에 대한 기억
    - to-do 주석문 등
- 하위 목표 라벨 붙이기
  - 문제를 어떤 작은 단계로 나눌 수 있는지 명시적으로 기록

#### 요약

- 프로그래밍을 할 때 검색, 이해, 전사, 증가, 탐구 등 다양한 프로그래밍 활동을 함께 수행함
- 프로그래밍 중 중단은 성가실 뿐만 아니라 코드에 대한 정신 모델을 재구성하는 데 시간이 걸리기 때문에 생산성에 악영향을 끼침
- 중단에 잘 대처하려면 정신 모델을 노트, 문서, 주석문으로 작성해야 함
- 작업을 완료할 수 없다면 계획을 문서화해서 미래 기억을 의도적으로 도와주어야 함
- 방해를 받더라도 가급적이면 인지 부하가 낮을 때 중단이 일어나도록 해야 함

## 대규모 시스템의 설계와 개선

### 코드베이스의 특성 조사

- 프로그래밍 언어를 논할 때 우리는 기술적 영역을 살펴본다
  - 패러다임(객체 지향, 함수형), 타입 시스텝 여부, 언어가 바이트코드로 컴파일되는지 또는 다른 언어로 된 프로그램에 의해 인터프리트되는지
  살펴봄

#### 인지적 차원

- 오류 경향성
  - 일부 프로그래밍 언어에서는 다른 언어보다 실수를 저지르기 쉽다
    - 자바스크립트는 인기가 많지만 몇몇 특이한 예외 사례가 존재
    - 동적 유형 언어에서는 변수가 생성될 때 데이터 타입이 정해지지 않기 때문에 런타임에 오류가 발생할 수 있음
  - 프로그래밍 언어가 아닌 코드베이스 역시 오류 경향성이 있다.
    - 일관성 없는 규칙, 문서 부족, 모호한 이름 등등 때문
  - 코드베이스는 프로그래밍 언어로부터 차원을 상속하는 경우도 있다.
    - 파이썬으로 작성된 모듈은 C로 작성된 유사한 라이브러리보다 오류가 발생하기 쉬움
    - 파이썬은 C처럼 오류를 잡을 수 있는 강한 타입 시스템이 없기 때문
- 일관성
  - 이름과 규약을 일관성 없이 사용하는 프레임워크나 언어는 인지 부하를 더 많이 가져올 수 있음
    - 두뇌가 그것이 무엇인지 이해하는 데 더 많은 에너지를 소모하고, 관련 정보를 찾는 데 더 많은 시간을 쓰기 때문
- 분산성
  - 프로그래밍 구성 요소가 얼마나 많은 공간을 차지하는지 나타냄
    - 파이썬의 for 루프와 C++의 for 루프는 코드의 줄 수에 차이가 있음
  - 코드의 줄 수 뿐 아니라 코드가 얼마나 많은 청크로 구성되어 있는지도 고려할 수 있음
- 숨겨진 의존성
  - 의존 항목이 사용자에게 어느 정도로 가시적으로 나타나는지를 보여줌
  - 숨겨진 의존성이 높은 시스템
    - HTML 페이지에 자바스크립트로 제어되는 버튼이 있고, 자바스크립트는 다른 파일에 저장되어 있는 경우
      - 자바스크립트 파일만 봐서는 그 함수가 어느 HTML 페이지로부터 호출되는지 알기 ㅇ어려움
  - 다른 함수나 클래스 안에서 어떤 함수가 호출되는지 아는 것과 모르는 경우 전자가 후자보다 더 쉽다
    - 주어진 함수를 어떤 클래스나 함수가 호출하는지 아는 것
  - IDE는 숨겨진 의존성을 보여주는 기능을 가지고 있지만, 의존 항목을 찾기 위해서는 여전히 클릭이나 단축키를 사용하는 추가적인 동작이 필요
  - 광범위한 문서로 숨겨진 의존성을 보완할 수 있음
- 잠정성
  - 도구를 사용하는 동안 생각하는 것이 얼마나 쉬운지에 대한 것
  - 코드베이스나 프로그래밍 언어가 매우 엄격하다면 코드를 사용해 생각을 표현하는 것이 어려움
    - 잠정성이 낮음
  - 초보자라면 애매한 아이디어와 불완전한 코드를 표현하는 것이 필요할 수 있기 땜누에 잠정성은 학습 능력의 피ㅓㄹ수적 요소
    - 코드 계획과 동시에 타입과 문법에 대해 생각하는 것은 많은 인지 부담
- 점도
  - 특정 시스템을 변경하는 것이 얼마나 어려운가에 대한 차원
  - 동적 타입 언어로 작성된 코드를 변경하는 것이 조금 더 쉬움
    - 코드만 변경하면 됨, 타입 정의를 변경할 필요가 없음
  - 코드를 변경하고 나서 컴파일하거나 테스트 실행에 시간이 오래 걸린다면, 그 변경에 대한 점도는 증가함
- 점진적 평가
  - 주어진 시스템에서 부분적인 작업을 확인하거나 실행하는 것이 얼마나 쉬운지에 대한 것
- 역할 표현력
  - 프로그램에서 여러 가지 다른 부분의 역할을 얼마나 쉽게 알 수 있는지를 나타냄
    - file.open()처럼 거의 모든 프로그래밍 언어에서 매개변수가 없는 함수를 호출할 때 여전히 끝에 두 개의 괄호를 가지고 있다는 점
    - 함수의 끝에 있는 괄호는 역할 표현력의 일례
    - 문법을 통해서도 역할 표현력이 나타낼 수 있음
      - 불리언 값을 반환하는 함수로 set이 아니라 is_set을 호출하면 이 코드를 읽을 때 변수의 역할을 이해하는 데 도움이 됨
- 매핑 근접성
  - 로그래밍 언어 또는 코드가 문제의 해결 영역에 얼마나 가까운지를 의미
  - ATL은 벡터 미적분과 매우 밀접한 관계
    - 해결하려는 문제가 벡터 미적푼을 사용해서 풀리는 경우에 사용하면 좋음
  - 코볼은 비즈니스 및 금융과 밀접한 관계를 가진 언어
  - 자바, 파이썬, 자바스크립트를 포함한 대부분의 현대 프로그래밍 언어는 어느 문제나 해결할 수 있는 범용적 언어
    - 매핑 근접성이 좋지 않음.
    - 항상 나쁜것은 아님
- 힘든 정신 활동
  - 사용자가 시스템의 외부에서 힘든 정신 활동을 수행할 때 생각하는 것이 매우 어려울 수 있음
  - 하스켈과 같은 언어는 사용자가 모든 함수와 매개변수의 타입을 고려해야 함
- 보조 표기법
  - 프로그래머가 공식 규격에는 없는 의미를 코드에 추가할 가능성
    - 소스 코드에 주석문을 추가할 가능성 등
- 추상화
  - 시스템의 사용자가 기본적으로 제공되는 추상화만큼 강력한 추상화를 만들 수 있는지 여부
  - 라이브러리 사용자가 추가 기능을 위해 하위 클래스를 만들 수 있도록 허용하는 것은 API 호출만 허용하는 라이브러리보다 추상화가 더 강력함
- 가시성
  - 시스템의 다른 부분을 얼마나 쉽게 볼 수 있는지를 나타냄
  - 코드가 여러 파일로 나뉘어 있는 경우 코드베이스가 어떤 클래스로 구성되어 있는지 확인이 어려움

### 차원 및 활동

#### 차원이 활동에 미치는 영향

- 검색
  - 숨겨진 종속성은 검색 활동에 악영향을 미칠 수 있음
    - 어떤 코드가 어디에서 호출되는지 모를 경우 그 다음 코드의 어느 부분을 읽어야 할지 알기 어려워 검색 속도가 느려지기 때문
  - 분산성은 코드를 더 길게 만들고 검색할 코드가 더 많아지기 때문에 부정적 영향
  - 보조 표기법은 주석문 및 변수 이름의 정보 위치를 나타낼 수 있기 때문에 긍정적 영향
- 이해
  - 코드베이스의 가시성이 낮으면 클래스와 기능의 상호관계를 파악하기 어려워 이해에 부정적 영향
  - 역할 표현력은 변수 및 기타 개체의 유형과 역할이 명확하면 이해하기 쉬워지기 때문에 긍정적 영향
- 전사(미리 정의된 계획을 기반으로 기능을 구현할 때)
  - 일관된 코드베이스는 이해하기 쉽지만 새로운 기능을 구현할 때는 코드베이스에 맞춰야 하므로 정신적 노력이 추가로 필요할 수 있기 때문에
  일관성은 부정적 영향을 끼칠 수 있음
    - 장기적으로는 그럴만한 가치가 있지만 추가로 기울여야 하는 노력인 것은 사실
- 증가
  - 새 기능을 추가할 때 코드베이스에서 프로그래밍 개념보다는 코드의 목표를 뚜렷이 알 수 있다면 새로운 코드를 추가하는 것이 더 쉽기 때문에
  매핑 근접성이 도움이 됨
  - 점도가 높은 코드베이스는 코드를 추가하기 어려움
- 탐구
  - 잠정성과 점진적 평가가 새로운 설계 아이디어를 탐구할 때 도움이 됨
  - 힘든 정신 활동과 추상화는 프로그래머에게 높은 인지 부하를 유발하기 때문에 악영향을 끼칠 수 있음

### 요약

- 대부분의 경우 서로 다른 차원 간의 트레이드오프가 이루어져야 함.
  - 한 차원을 개선하면 다른 차원이 감소할 수 있음
- 코드베이스가 최적화할 대상인 차원에 대한 요구 사항은 수행하려는 활동에 따라 다름

## 새로운 개발자 팀원의 적응 지원

- 상급 개발자라면 후배들이 더 효과적으로 학습할 수 있기 위해서 그들이 경험하는 인지 부하를 관리해야 함

### 적응 지원의 문제

- 많은 프로그래머가 교수법이나 멘토링에 대한 교육을 받은 것은 아니기 때문에 적응 지원 과정이 양쪽 모두에게 좌절감을 줄 수 있음
- 적응 지원은 대체로 다음과 같은 과정으로 진행
  - 선임 개발자가 새 팀원에게 많은 정보를 줌. 정보의 양이 너무 많아서 높은 인지 부하를 유발
    - 팀원들, 코드베이스의 도메인, 워크플로, 코드베이스를 한꺼번에 소개
  - 소개가 끝난 후 선임 개발자는 새 팀원에게 질문을 하거나 과제를 줌
    - 선임 개발자는 이것을 아주 간단한 일로 여김, 작은 버그를 고치거나 작은 기능을 추가하는 것
  - 도메인이나 프로그래밍 언어 혹은 두 가지 모두 관련 청크의 부족과 관련 자동화 기술 부족으로 인해 인지 부하가 높아지고 새 팀원은 적응에 어려움을 겪음
- 새 팀원은 작업 기억 공간에 과부하가 걸리기 때문에 새로운 코드베이스에서 효과적으로 프로그래밍을 할 수 없고, 새롱누 정보를 제대로 유지할 수도 없음
  - 양쪽 모두에게 좌절감을 안겨주고 서로에 대한 잘못된 가정으로 이어짐
    - 선임 개발자는 팀원이 똑똑하지 않나?
    - 새 팀원은 프로젝트가 매우 어렵네
- 선임자들이 효과적으로 가르치고 설명하는 데 어려움을 겪는 이유 중 하나는 많은 경우 전문가의 저주 때문임
  - 어떤 기술을 충분히 익히고 나면, 그 기술이나 지식을 배우는 것이 얼마나 어려웠는지 잊어버림
  - 새 팀원이 동시에 처리할 수 있는 새로운 작업의 수를 과대평가함

### 적응 지원 개선

#### 작업은 하나의 프로그래밍 활동으로만 제한

- 사람들이 코드베이스에서 수행할 수 있는 5가지 활동(전사, 탐구, 이해, 검색, 증가)
- 적응 지원 과정의 문제 중 하나는 새 팀원에게 최소 네 가지 서로 다른 활동을 요구함
  - 기능을 구현할 수 있는 적절한 장소 또는 관련 정보 검색, 새로운 소스 코드의 이해, 코드베이스 탐구, 새로운 기능으로 코드베이스 수정
- 적응 지원 기간에는 다섯 가지 범주 각각에 해당하는 활동을 선택하고 새 팀원에게 하나씩 시키는 것이 가장 좋음
  - 탐구
    - 코드베이스의 전체적인 이해를 위한 코드 훑어보기
  - 검색
    - 특정 인터페이스를 구현한 클래스 찾기
  - 전사
    - 구현할 메서드에 대한 명확한 계획을 알려주기
  - 이해
    - 코드의 여러 측면에 대해 이해하기, 특정 메서드 요약
  - 증가
    - 향후 계획을 포함해서 기존 클래스에 한 가지 기능을 추가하기

#### 새 팀원의 기억 지원

- LTM 지원: 관련 정보 설명
  - 코드베이스 작업 시 중요한 역할을 하는 관련 정보를 숙지해둠으로써 새로운 사용자를 위한 적응 지원 과정을 준비할 수 있음
  - 새 팀원이 도착하기 전에 프로젝트를 종료한 팀에서 할 수 있음
  - 예시) 코드에서 발생할 수 있는 모든 중요한 도메인 개념을 문서화
    - 웹 애플리케이션에 라라벨을 사용하고 젠킨스로 빌드해 히로쿠에 배포한다
    - 같은 문장은 하나라도 알지 못한다면 문장을 이해하기 어려움
- STM 지원: 규모가 작고 집중할 수 있는 작업의 준비
  - 새 팀원은 여러 가지 프로그래밍 활동을 하도록 요청 받고, STM에 과부하가 유발됨

#### 코드 함께 읽기

- 활성화
  - 관련된 것들을 적극적으로 생각해서 이미 가지고 있는 지식을 활성화하는 것
- 모니터링
  - 텍스트를 읽으면서 자신이 이해한 것(이해하지 못한것도)을 관찰하고 기록하는 것
- 중요도 결정
  - 텍스트에서 어느 부분이 중요한지 결정하는 것
- 추론
  - 텍스트에서 명시적으로 주어지지 않은 사실을 유추하는 것
- 시각화
  - 깊이 있는 이해를 위해 텍스트에 대한 도표를 만드는 것
- 질문
  - 텍스트에 질문하는 것
- 요약
  - 텍스트를 짧게 요약하는 것

### 요약

- 전문가와 초보자는 다르게 생각하고 행동함
  - 전문가는 코드에 대해 추상적으로 추론할 수 있고 코드 자체를 언급하지 않고도 코드에 대해 생각할 능력을 가지고 있음
  - 초보자는 코드의 세부 사항에 집중하는 경향이 있고 세부 사항에서 벗어나는 데 어려움을 겪음
- 중급 프로그래머들도 새로운 정보를 배우다 보면 초급 수준의 사고방식으로 떨어질 때도 있음
- 새로운 개념을 배우는 사람들은 추상적인 용어와 구체적인 예를 모두 배울 필요가 있음
- 새로운 개념을 배우는 사람들은 새로 배운 개념을 기존 지식과 연결할 시간이 필요함
- 적응 지원 과정에서 새 팀원이 수행할 프로그래밍 활동은 한 번에 한 개로 제한해야 함
